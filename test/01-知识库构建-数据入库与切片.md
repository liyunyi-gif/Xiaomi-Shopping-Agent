# 测试用例 · 知识库构建（数据入库与切片）

> 模块：Knowledge 子节点 · ingest 管线
> 前缀：`INGEST`
> 覆盖验收点：架构.md §3.2 Knowledge 入库构建；技术架构 知识库Agent §2（Tika 解析 → 切片 → embedding 入库）
> 落点表：`t_knowledge_base` / `t_knowledge_document` / `t_knowledge_chunk` / `t_knowledge_vector`

---

## 验收目标

1. 多格式文档（PDF/Word/HTML/TXT）能被正确解析为纯文本。
2. 切片策略合理：按 token 上限切，切片间有重叠，不切断语义单元（标题/段落）。
3. 切片文本与向量分离存储（`t_knowledge_chunk` 存文本+tsv，`t_knowledge_vector` 存向量 HNSW）。
4. 关键词路的 `tsv` 列由触发器自动维护，可被 `plainto_tsquery` 命中。
5. 重复入库幂等，不产生重复切片。
6. 元数据（title/spec_text 等字段）落库，供 rerank 字段加权。

---

## INGEST-001 PDF 商品资料解析为纯文本（正向）

- 模块：知识库构建·文档解析
- 用例类型：正向
- 优先级：高
- 前置条件：上传一份小米14 商品规格 PDF（含中文）
- 测试步骤：
  1. 调用 ingest 接口，传入 PDF 文件流
  2. Tika 解析
- 预期结果：
  - 解析出的纯文本不含 PDF 控制字符/乱码
  - 关键规格（如"徕卡""5000万像素"）在文本中可检索到
- 需求来源：架构.md §3.2；技术架构 知识库Agent §2

## INGEST-002 多格式文档解析（Word/HTML/TXT）

- 模块：知识库构建·文档解析
- 用例类型：正向
- 优先级：中

  - 三种格式解析出的核心文本一致（去除标签后）

## INGEST-003 切片不超 token 上限且含重叠（边界）

- 模块：知识库构建·切片
- 用例类型：边界
- 优先级：高
- 前置条件：解析得到一段 3000 token 的长文本，配置 chunk_size=512、overlap=64
- 测试步骤：
  1. 执行切片
  2. 统计每个切片 token 数与相邻切片首尾重叠
- 预期结果：
  - 每个切片 token 数 ≤ 512
  - 相邻切片存在 ≥64 token 重叠
  - 切片拼接后内容无损（顺序可还原）
- 需求来源：技术架构 知识库Agent §2

## INGEST-005 切片文本与向量分离存储（正向）

- 模块：知识库构建·入库
- 用例类型：正向
- 优先级：高
- 前置条件：一次完整入库（解析→切片→embedding）
- 测试步骤：
  1. 查 `t_knowledge_chunk`：应写入 N 条切片，含 content / title / spec_text / tsv
  2. 查 `t_knowledge_vector`：应写入 N 条向量，通过 `chunk_id` 回链切片
- 预期结果：
  - chunk 与 vector 一一对应（chunk_id 外键完整）
  - vector.embedding 维度 = 配置维度（1024）
  - 文本字段不在 vector 表（分离）
- 需求来源：技术架构总纲 §5.1（向量与文本分离）

## INGEST-006 tsv 列由触发器自动维护（正向）

- 模块：知识库构建·全文检索索引
- 用例类型：正向
- 优先级：高
- 前置条件：已完成 INGEST-005 入库
- 测试步骤：
  1. 直接 INSERT 一条 chunk（不手写 tsv）
  2. 用 `plainto_tsquery('徕卡')` 查询
- 预期结果：
  - 触发器自动生成 tsv，无需手动维护
  - 查询能命中含"徕卡"的切片
- 需求来源：技术架构总纲 §5.1（GIN(tsv) + 触发器维护）

- 模块：知识库构建·文档解析
- 用例类型：异常
- 优先级：中
