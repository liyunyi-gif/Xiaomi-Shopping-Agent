## 0. 测试设计总则

### 0.1 测试对象边界

| 节点 | 是否测试 | 说明 |
|---|---|---|
| **Orchestrator 主 Agent** | ✅ 重点 | 唯一真 Agent：意图识别、质量判断、重检循环、三层记忆、跨节点编排 |
| **Knowledge 子节点** | ✅ 重点 | 无状态检索管线：入库/切片、查询重写、双路召回、rerank、置信信号 |
| **Shopping 子节点** | ✅ 重点 | 无状态 MCP 网关：工具编排、状态信号 |
| **MCP Server** | ✅ | 外部服务→MCP 工具封装，工具契约 |
| **common 契约** | ✅ | 跨节点 DTO 字段约束、序列化 |

### 0.2 用例编号规则

- 格式：`模块前缀-NNN`（如 `INGEST-001`、`INTENT-014`、`JUDGE-005`）
- 模块前缀见下方 §2 模块清单

### 0.3 用例统一字段模板

每条用例必须包含：

```
### XXX-NNN 用例标题
- 模块：xxx
- 用例类型：正向 / 异常 / 边界
- 优先级：高 / 中 / 低
- 前置条件：xxx
- 测试步骤：
  1. ...
  2. ...
- 预期结果：
  - 断言1（可量化）
  - 断言2
- 需求来源：架构.md §X / 原则 Pn
```

### 0.4 优先级口径（与数据库设计.md 一致）

- **高** = 核心闭环必需（不通过则主流程跑不通）
- **中** = 业务完整度
- **低** = 可观测/增强

## 1. 验收点清单（Coverage Backbone）

> 全部验收点从 架构.md 提取，每项至少 1 条用例命中。下方 [✓] 表示已有用例覆盖。

### 1.1 七大设计原则（架构.md §2）

| 编号 | 原则 | 验收点 | 覆盖用例 |
|---|---|---|---|
| P1 | 意图识别唯一入口 | 只有主 Agent 做意图识别，子节点不做 | INTENT-001, ARCH-001 |
| P2 | 唯一开口权 | 只有主 Agent 对用户说话，子节点永不直接交互 | MCP-005, MEM-001 |
| P3 | 记忆单点 | 只有主 Agent 持有记忆，子节点完全无状态 | MEM-002, ARCH-002 |
| P4 | 子节点只举手不开口 | 子节点返回结构化信号，不直接问用户 | MCP-003, MCP-004 |
| P5 | 能力正交 | 检索型归 Knowledge，外部调用型归 Shopping，不越界 | ARCH-003, RECALL-001 |
| P6 | 质量判断上移 | 检索质量由主 Agent 判定，不由 Knowledge 自判 | JUDGE-001, RECALL-006 |
| P7 | 克制原则 | 重检有上限、必须换策略、达上限退化坦诚告知 | JUDGE-006, JUDGE-007 |

### 1.2 三节点职责（架构.md §3）

| 职责 | 验收点 | 覆盖用例 |
|---|---|---|
| 主 Agent · 意图识别（三类一级意图） | KNOWLEDGE/TOOL/SYSTEM 正确分类 | INTENT-002~004 |
| 主 Agent · 澄清权（意图级+槽位级） | 低置信/缺槽位触发主 Agent 反问 | INTENT-005, MCP-004 |
| 主 Agent · 质量判断 + 循环 | 三信号4步判定 + 重检状态机 | JUDGE-001~007 |
| 主 Agent · 三层记忆 | 短期/长期/存档 分层与注入 | MEM-001~008 |
| Knowledge · 入库构建 | Tika 解析→切片→向量入库 | INGEST-001~008 |
| Knowledge · 查询重写/拆分 | 指代消解、口语化转术语、子问题拆分 | RECALL-002, RECALL-003 |
| Knowledge · 双路召回 | 语义路 + 关键词路 并行召回 | RECALL-001, RECALL-004 |
| Knowledge · rerank | 加权打分，Top-1 精准 | RECALL-005 |
| Knowledge · 返回置信信号 | 三信号：相关度/命中实体/召回数 | RECALL-006 |
| Shopping · 工具编排 | 按主 Agent 指令编排加购/下单 | MCP-001, MCP-002 |
| Shopping · 状态信号 | 成功/需澄清/失败 + 缺失槽位清单 | MCP-003, MCP-004 |

### 1.3 核心数据流（架构.md §4）

| 数据流 | 验收点 | 覆盖用例 |
|---|---|---|
| §4.1 知识问答流（含一次重检） | 充分→直出；不够→换策略重检→充分 | FLOW-001, JUDGE-005 |
| §4.2 购物执行流 | 加购指令→Shopping 编排→转述 | FLOW-002, MCP-001 |
| §4.3 子节点举手流 | 缺参数→返回 NEED_CLARIFY→主 Agent 开口 | FLOW-003, MCP-004 |
| §4.4 混合意图流 | 主 Agent 串联两个子节点 | FLOW-004 |

### 1.4 检索质量判断（架构.md §5）★ 简历重点

| 验收点 | 覆盖用例 |
|---|---|
| 三信号4步判定（失败/不全/不够/充分） | JUDGE-001~004 |
| 重检循环状态机 | JUDGE-005, JUDGE-006 |
| 换策略重检（禁原样重查） | JUDGE-007 |
| 达上限退化坦诚告知（防幻觉） | JUDGE-008 |

### 1.5 三层记忆（架构.md §6）★ 简历重点

| 验收点 | 覆盖用例 |
|---|---|
| 对话存档 ≠ 记忆（分层不混存） | MEM-001 |
| ①短期记忆（Redis，最近N轮） | MEM-003 |
| ②长期记忆（PgSQL提炼，跨会话） | MEM-004, MEM-005 |
| ③对话存档（PgSQL逐字，不进默认上下文） | MEM-006 |
| 加载注入=②+①，存档③不进上下文 | MEM-007 |
| 会话结束批量提炼 | MEM-005 |
| 遗忘：过期淘汰 + 用户手动清除 | MEM-008 |

### 1.6 主↔子节点信息契约（架构.md §8）

| 契约 | 验收点 | 覆盖用例 |
|---|---|---|
| 主→Knowledge 注入（意图+问题+快照） | 字段完整、无主观自评 | CONTRACT-001, RECALL-006 |
| Knowledge 返回（结果+三信号） | 不含主观质量自评 | CONTRACT-002 |
| 主→Shopping 注入（意图+槽位+快照） | 槽位已澄清 | CONTRACT-003 |
| Shopping 返回（状态+结果+缺失槽位） | NEED_CLARIFY 必带缺失清单 | CONTRACT-004 |
| 会话快照字段 | userId/intent/cart/history 齐全 | CONTRACT-005 |

## 4. 测试环境与数据准备

### 4.1 依赖中间件

| 中间件 | 版本 | 用途 | 地址 |
|---|---|---|---|
| PostgreSQL + pgvector | 15 / 0.7+ | 结构化数据 + 向量 | 192.168.122.128:5432 |
| Redis | 7.x | 短期记忆 | 192.168.122.128:6379 |
| MCP Server | - | 外部工具进程 | localhost:8090 |

### 4.2 测试数据集

- **商品知识库**：从小米商城抽取 10~20 条真实 SKU 资料（PDF/HTML），覆盖手机/平板/穿戴品类
- **意图样本**：每类一级意图各 20+ 条 query（含口语化/指代/多实体）
- **边界 query**：空输入、超长、纯符号、混合意图

### 4.3 Mock 策略

| 外部依赖 | 策略 |
|---|---|
| 通义 LLM（意图识别/重写/生成/提炼） | Mock `ChatClient`，返回固定 JSON |
| 通义 Embedding | Mock，返回固定 1024 维向量 |
| MCP Server（真实外部业务系统） | Mock `ToolCallbackProvider` |
| Redis / PostgreSQL | 用 Testcontainers（推荐）或嵌入式 |

## 5. 执行说明

1. 本目录是**用例设计**（Markdown），编码阶段据此在各自模块 `src/test/java` 下编写 JUnit 5 测试。
2. 每个 `.md` 文件 ↔ 一个测试类，每个用例编号 ↔ 一个 `@Test` 方法。
3. 代码完成后用 **JaCoCo** 跑覆盖率，目标：指令覆盖率 ≥ 95%。
4. 用例中发现设计缺陷，回头修订 架构.md（以逻辑设计为准）。
